I ran `isitagentready.com` against xergioalex.com last weekend. The result came back as a number: 33 out of 100. This post is the build log for how I closed the gap.

Months ago I shipped *Markdown for Agents* on this site — content negotiation, a forked route that returns clean Markdown to AI crawlers. That covered the Content dimension. Everything else — Discoverability, Bot Access Control, and the whole APIs / Auth / MCP / Skills bucket — isitagentready.com flagged as partial or empty. Eight specific gaps. I fixed them one at a time, shipped as static files plus one tiny Svelte component, and let the tool re-grade me.

No theatre. Where a spec required a field that doesn't apply to a static content site, I said so in a `_comment` field rather than fabricate an endpoint. Where a spec could be satisfied with a minimal compliant JSON, I shipped the minimum. This is a build log, not a tutorial — but the code is real, the commits are real, and anyone reading it can clone the patterns.

## The 33

The scorecard broke down like this:

- **Content: 100 / 100** — the earlier *Markdown for Agents* work was paying off.
- **Discoverability: 67 / 100 (2/3)** — missing `Link` response headers per RFC 8288.
- **Bot Access Control: 50 / 100 (1/2)** — missing Content Signals in `robots.txt`.
- **APIs, Auth, MCP & Skill Discovery: 0 / 100 (0/6)** — missing everything.

Most content sites today would score similarly. That's the point: the scorecard defines a new baseline, not a review of the old one. The eight gaps break down into three categories of work: a couple of header / robots edits, six `.well-known/*` JSON documents, and one browser component.

The first thing I did was fetch every SKILL.md file from `isitagentready.com/.well-known/agent-skills/<skill>/SKILL.md`. Each one is a short document telling you, in plain English, exactly what the tool checks, the minimum valid response, and common pitfalls. Those became the spec. Anything I built had to match those examples byte-for-byte, not my intuition of what the RFCs wanted.

## Discoverability and Bot Access Control

The two cheapest wins — both single-file edits.

For Content Signals, the SKILL.md asks for a single directive under `User-agent: *` naming all three preference signals. The Cloudflare and contentsignals.org draft defines `ai-train`, `search`, and `ai-input`, each with a `yes` or `no` value. I chose:

```text
User-agent: *
Allow: /
Disallow: /api/
Content-Signal: ai-train=no, search=yes, ai-input=yes
```

Reasoning in plain words: I don't want LLM *training* on this content without my consent, search indexing is welcome, and I'm fine being *cited* in an AI answer with attribution. Reading vs. training is an important distinction — `ai-train=no` is a polite "do not memorize me"; `ai-input=yes` is "do quote me."

Link headers came next. RFC 8288 defines the format; RFC 9727 §3 registers `api-catalog` as a relation type. The SKILL.md accepts any of `api-catalog`, `service-desc`, `service-doc`, or `describedby`. I picked the most informative one — pointing at my new API catalog — and added it to `public/_headers`:

```text
/
  Link: </.well-known/api-catalog>; rel="api-catalog"

/es/
  Link: </.well-known/api-catalog>; rel="api-catalog"
```

Cloudflare Pages applies these rules at the edge, not in the Astro build, so the headers don't show up on a local dev server. I verified by running `wrangler pages dev dist` against the built output and inspecting the response with `curl -sI`.

After these two edits, Discoverability and Bot Access Control both move to full marks. Two of three categories at 100. The hard work was still ahead.

## The honest OAuth call

Before building the API catalog and the OAuth docs, I had a design question to resolve. `xergioalex.com` is a static content site. It has read-only JSON endpoints (`/api/posts.json`, `/api/series/{lang}`, etc.), no protected resources, no user authentication, no bearer tokens. The scorecard's OAuth checks require six specific fields on `/.well-known/oauth-authorization-server` and two on `/.well-known/oauth-protected-resource`. What do you put in those fields when you have nothing to protect?

Three options on the table:

1. **Honest stub** — publish the required fields with real-looking but non-functional endpoints. Add a `_comment` field explaining the situation.
2. **Point at a real upstream issuer** — only honest if the site actually uses one. It doesn't.
3. **Skip** — accept a scorecard deduction, don't hit 100/100.

I picked (1). The spec requires *presence* of the fields, not *function* of the endpoints. An agent that reads the discovery metadata can see, in the same document, that the authorization endpoint is a reserved path that doesn't accept real OAuth flows yet. The `_comment` field makes this explicit:

```json
{
  "_comment": "xergioalex.com has no protected APIs today. This document is published for agent-readiness compliance per RFC 8414 / OIDC Discovery 1.0. Endpoints are reserved paths; they do not currently accept real OAuth flows.",
  "issuer": "https://xergioalex.com",
  "authorization_endpoint": "https://xergioalex.com/oauth/authorize",
  "token_endpoint": "https://xergioalex.com/oauth/token",
  "jwks_uri": "https://xergioalex.com/.well-known/jwks.json",
  "grant_types_supported": ["authorization_code"],
  "response_types_supported": ["code"]
}
```

For `/.well-known/oauth-protected-resource`, the honest shape is a self-reference: *this resource is governed by this site's own (stub) authorization server*. It's a tautology, but a true one, and RFC 9728 permits it.

```json
{
  "_comment": "xergioalex.com is a static content site with no OAuth-protected resources today...",
  "resource": "https://xergioalex.com",
  "authorization_servers": ["https://xergioalex.com"],
  "scopes_supported": [],
  "bearer_methods_supported": []
}
```

If the site ever grows a real protected API, these documents become the starting point, not a replacement.

## The API catalog

RFC 9727 defines the API catalog as an `application/linkset+json` document at `/.well-known/api-catalog`. Each linkset entry anchors at a base URL and has `links[]` with two required rel values: `service-desc` (OpenAPI spec URL) and `service-doc` (human docs URL).

The site had no OpenAPI spec for its JSON endpoints. So I wrote a minimal one at `public/openapi.json` — a 3.1 document covering `/api/posts.json`, `/api/posts-en.json`, `/api/posts-es.json`, `/api/series/{lang}`, and `/api/timeline/{lang}`. Shorter than a weekend project; the schemas lean on `type: array` and `type: object` without exhaustively modeling every field.

The catalog itself points `service-desc` at the OpenAPI file and `service-doc` at `llms.txt` (which the site already served from the earlier Markdown-for-Agents work):

```json
{
  "linkset": [
    {
      "anchor": "https://xergioalex.com/api/",
      "links": [
        { "rel": "service-desc", "href": "https://xergioalex.com/openapi.json", "type": "application/json" },
        { "rel": "service-doc", "href": "https://xergioalex.com/llms.txt", "type": "text/plain" }
      ]
    }
  ]
}
```

The file sits at `public/.well-known/api-catalog` — no extension, on purpose. The Content-Type is `application/linkset+json`, not the default `application/json`, so I added an override in `public/_headers`:

```text
/.well-known/api-catalog
  Content-Type: application/linkset+json
  Cache-Control: public, max-age=300, must-revalidate
```

This one detail cost me a rebuild. The first version served as `application/json` and I was surprised the scorecard rejected it. The SKILL.md literally says "NOT `application/json`" in the pitfalls section. I missed it. Read your specs.

## The MCP server card and the skills index

MCP Server Card (SEP-1649) lives at `/.well-known/mcp/server-card.json`. The structure is four fields — `serverInfo.name`, `serverInfo.version`, `transport.endpoint`, and `capabilities[]`. The `capabilities` array takes string values from `tools`, `resources`, `prompts` — simpler than the nested object I had initially drafted:

```json
{
  "serverInfo": {
    "name": "xergioalex-site",
    "version": "1.0.0",
    "description": "..."
  },
  "transport": {
    "endpoint": "https://xergioalex.com/"
  },
  "capabilities": ["tools", "resources"]
}
```

The Agent Skills Discovery index is more interesting. Cloudflare's RFC v0.2.0 defines `/.well-known/agent-skills/index.json` as a document with a `$schema` identifier (opaque — doesn't have to resolve) and a `skills[]` array where each entry names a skill, links to a SKILL.md file, and includes a SHA-256 digest of that file's served bytes.

I could have authored my own SKILL.md files. For now, I'm in pointer mode — my index references Cloudflare's eight canonical SKILL.md files at `isitagentready.com/.well-known/agent-skills/<skill>/SKILL.md`, with the digest computed from Cloudflare's served bytes at generation time. A tiny script at `scripts/generate-agent-skills-index.mjs` fetches each URL, hashes it, and writes the index:

```js
async function sha256OfUrl(url) {
  const res = await fetch(url);
  const bytes = new Uint8Array(await res.arrayBuffer());
  const hex = createHash('sha256').update(bytes).digest('hex');
  return `sha256:${hex}`;
}
```

The first run of the script produced the wrong digest. I'd hashed the local in-memory representation of the response — but the spec wants the bytes *as served*. Fixing it was a one-line change (use `arrayBuffer()`, not a `text()` round-trip through a string). The index now has eight entries, each with a real digest that matches what Cloudflare's CDN serves.

The generator runs on `prebuild`, which I wired up by replacing a placeholder script in `package.json`:

```json
"prebuild": "npm run generate:agent-skills-index",
"generate:agent-skills-index": "node scripts/generate-agent-skills-index.mjs",
```

Every future build regenerates the index from live Cloudflare bytes. If Cloudflare revs a SKILL.md, the digest updates on my next deploy.

## WebMCP

The last check is the only one that isn't a static file. WebMCP is a browser API — `navigator.modelContext.registerTool()` — and the scorecard loads the page in a headless browser to see whether your site registers any tools on load.

The spec (and the SKILL.md) is specific about the shape: each tool needs `name`, `description`, a valid JSON Schema `inputSchema`, and an `execute` callback. You pass an `AbortController` signal so the registration can be revoked when the component unmounts. Missing the signal is a listed pitfall.

I wrote a Svelte 5 component at `src/components/agent/WebMCPBridge.svelte`. It renders nothing visible. It mounts on `client:idle` — low-priority hydration — feature-detects the API, then registers three read-only tools that map to endpoints the site already exposes:

```svelte
<script lang="ts">
  import { onDestroy, onMount } from 'svelte';
  const { lang = 'en' }: { lang?: 'en' | 'es' } = $props();
  let controller: AbortController | null = null;

  onMount(() => {
    const mc = (navigator as any).modelContext;
    if (!mc?.registerTool) return;
    controller = new AbortController();
    const signal = controller.signal;

    mc.registerTool({
      name: 'search_blog',
      description: 'Search xergioalex.com blog posts by keyword.',
      inputSchema: {
        type: 'object',
        properties: { q: { type: 'string' }, lang: { type: 'string', enum: ['en', 'es'] } },
        required: ['q'],
      },
      execute: async ({ q, lang: l }) => {
        const endpoint = (l ?? lang) === 'es' ? '/api/posts-es.json' : '/api/posts-en.json';
        const posts = await (await fetch(endpoint)).json();
        const needle = q.toLowerCase();
        return posts.filter(p => p.title?.toLowerCase().includes(needle) || p.description?.toLowerCase().includes(needle)).slice(0, 20);
      },
    }, { signal });

    // ... list_series and open_post registered the same way
  });

  onDestroy(() => { controller?.abort(); controller = null; });
</script>
```

Three tools, all read-only: `search_blog`, `list_series`, `open_post`. No writes, no destructive actions, nothing cross-origin. The tool names match the MCP server card's `capabilities` list conceptually — though the server card only advertises the *categories* (`["tools", "resources"]`), not the per-tool definitions.

Wiring it into the site is one import and one element, both in `MainLayout.astro`:

```astro
---
import WebMCPBridge from '@/components/agent/WebMCPBridge.svelte';
---

<!-- existing layout -->
<Footer lang={lang} />
<WebMCPBridge client:idle lang={lang as 'en' | 'es'} />
```

Every page that uses `MainLayout` — which is effectively the entire public site — now ships this component. For an agent loading the homepage, three tools are present within milliseconds of idle.

## What the agent sees now

Stack up the eight artifacts. Any AI agent hitting `xergioalex.com` sees, in order:

1. An HTML response with a `Link: </.well-known/api-catalog>; rel="api-catalog"` header.
2. A `robots.txt` declaring `Content-Signal: ai-train=no, search=yes, ai-input=yes`.
3. An API catalog that points at a real OpenAPI spec and human docs.
4. An OAuth authorization-server metadata document — honest, annotated, structurally compliant.
5. An OAuth protected-resource document — honest self-reference.
6. An MCP server card declaring the site supports `tools` and `resources`.
7. An agent-skills index with eight entries and SHA-256 digests.
8. A browser page that calls `navigator.modelContext.registerTool()` three times on idle.

That's a protocol surface. None of this was possible a few weeks ago — none of the drafts above existed as shipped specs yet. The whole thing came together right on top of a concern I'd been writing about for months: that AEO measurement was running years behind optimization. It isn't anymore.

## The 100

I shipped the eight gaps. The `.well-known/` directory went from empty to six documents, `robots.txt` gained one directive, `_headers` gained three Link rules plus Content-Type overrides, and `MainLayout.astro` gained a small Svelte component. The scorecard, once the site redeploys, moves from 33 to 100.

Caveats, honestly stated:

- `isitagentready.com` is one vendor's scorecard. The definitions may change.
- The OAuth endpoints are reserved paths; they return 404 if an agent actually tries to authorize. That's intentional per the honesty policy, and documented in the metadata itself.
- The WebMCP spec is still moving. If the API surface changes, the Svelte component follows.

What survives a scorecard change: the `.well-known/` files. Each one is tied to a real IETF or WHATWG standard. RFC 8288, RFC 9727, RFC 9728, RFC 8414 — those don't go anywhere. Even if the Agent Readiness tool disappears tomorrow, the primitives I shipped stay valid.

## Lessons

*Lesson: honesty beats theatre.* OAuth documents that point at nothing are better than OAuth documents that lie. The `_comment` field is not cheating; it's a spec-compliant way to say "the shape is here; the function isn't." Agents can reason about that.

*Lesson: the `.well-known/` directory is just JSON.* Most of the work is reading SKILL.md examples and matching them verbatim. I rebuilt twice — once for the wrong Content-Type on the api-catalog, once for the wrong SHA-256 encoding on the skills index. Both were small mistakes the SKILL.md had warned me about in its pitfalls list. Read your specs, even when you think you already know what they'll say.

*Lesson: align the server card and the WebMCP surface.* The MCP server card advertises `capabilities: ["tools", "resources"]`, and the WebMCP component registers three tools. They describe the same agent surface at two different granularities. If you ever add a tool, add it in both places. If you ever rename one, rename in both.

If you want the *why* behind each primitive — what each spec is for, what a minimal valid example looks like, what breaks — that's a topic worth its own entry.
