Corrí `isitagentready.com` contra xergioalex.com el fin de semana pasado. El resultado fue un número: 33 de 100. Este post es la bitácora técnica de cómo cerré la brecha.

Hace meses envié *Markdown for Agents* en este sitio — negociación de contenido, una ruta bifurcada que devuelve Markdown limpio a los crawlers de IA. Eso cubría la dimensión de Contenido. Todo lo demás — Descubribilidad, Control de Acceso de Bots y la categoría entera de APIs / Auth / MCP / Skills — isitagentready.com lo marcó como parcial o vacío. Ocho brechas específicas. Las arreglé una por una, las envié como archivos estáticos más un componente Svelte pequeño, y dejé que la herramienta me volviera a calificar.

Sin teatro. Donde un spec requería un campo que no aplicaba a un sitio de contenido estático, lo dije en un campo `_comment` en lugar de fabricar un endpoint. Donde un spec se podía satisfacer con un JSON mínimo compatible, envié el mínimo. Esto es una bitácora técnica, no un tutorial — pero el código es real, los commits son reales, y cualquiera que lea puede clonar los patrones.

## El 33

El tablero se desglosaba así:

- **Contenido: 100 / 100** — el trabajo anterior de *Markdown for Agents* estaba dando frutos.
- **Descubribilidad: 67 / 100 (2/3)** — faltaban headers `Link` de respuesta según RFC 8288.
- **Control de Acceso de Bots: 50 / 100 (1/2)** — faltaban Content Signals en `robots.txt`.
- **APIs, Auth, MCP y Skill Discovery: 0 / 100 (0/6)** — faltaba todo.

La mayoría de sitios de contenido hoy tendrían un puntaje similar. Ese es el punto: el tablero define una nueva línea base, no una revisión de la anterior. Las ocho brechas se dividen en tres categorías de trabajo: un par de ediciones de headers / robots, seis documentos JSON `.well-known/*`, y un componente de navegador.

Lo primero que hice fue traer cada archivo SKILL.md desde `isitagentready.com/.well-known/agent-skills/<skill>/SKILL.md`. Cada uno es un documento corto que te dice, en inglés claro, exactamente qué revisa la herramienta, la respuesta mínima válida y las trampas comunes. Esos se volvieron el spec. Cualquier cosa que construí tenía que coincidir con esos ejemplos byte por byte, no con mi intuición de lo que los RFCs querían.

## Descubribilidad y Control de Acceso de Bots

Las dos ganancias más baratas — ambas ediciones de un solo archivo.

Para Content Signals, el SKILL.md pide una sola directiva bajo `User-agent: *` nombrando las tres señales de preferencia. El draft de Cloudflare y contentsignals.org define `ai-train`, `search` y `ai-input`, cada una con un valor `yes` o `no`. Escogí:

```text
User-agent: *
Allow: /
Disallow: /api/
Content-Signal: ai-train=no, search=yes, ai-input=yes
```

La razón en palabras simples: no quiero *entrenamiento* de LLMs sobre este contenido sin mi consentimiento, la indexación en buscadores es bienvenida, y está bien ser *citado* en una respuesta de IA con atribución. Leer vs. entrenar es una distinción importante — `ai-train=no` es un cortés "no me memorices"; `ai-input=yes` es "sí cítame".

Luego vinieron los Link headers. RFC 8288 define el formato; RFC 9727 §3 registra `api-catalog` como tipo de relación. El SKILL.md acepta cualquiera de `api-catalog`, `service-desc`, `service-doc` o `describedby`. Escogí el más informativo — apuntando a mi nuevo API catalog — y lo agregué a `public/_headers`:

```text
/
  Link: </.well-known/api-catalog>; rel="api-catalog"

/es/
  Link: </.well-known/api-catalog>; rel="api-catalog"
```

Cloudflare Pages aplica estas reglas en el edge, no en el build de Astro, así que los headers no aparecen en un dev server local. Verifiqué corriendo `wrangler pages dev dist` contra el build y revisando la respuesta con `curl -sI`.

Después de estas dos ediciones, Descubribilidad y Control de Acceso de Bots pasan a 100. Dos de tres categorías al máximo. El trabajo duro todavía estaba por delante.

## La decisión honesta sobre OAuth

Antes de construir el API catalog y los docs de OAuth, tenía una pregunta de diseño que resolver. `xergioalex.com` es un sitio de contenido estático. Tiene endpoints JSON de solo lectura (`/api/posts.json`, `/api/series/{lang}`, etc.), sin recursos protegidos, sin autenticación de usuarios, sin bearer tokens. Las revisiones de OAuth del tablero requieren seis campos específicos en `/.well-known/oauth-authorization-server` y dos en `/.well-known/oauth-protected-resource`. ¿Qué pones en esos campos cuando no tienes nada que proteger?

Tres opciones sobre la mesa:

1. **Stub honesto** — publicar los campos requeridos con endpoints que parecen reales pero no son funcionales. Agregar un campo `_comment` explicando la situación.
2. **Apuntar a un issuer upstream real** — solo honesto si el sitio realmente usa uno. No lo hace.
3. **Saltar** — aceptar una deducción en el tablero, no llegar a 100/100.

Escogí la (1). El spec requiere *presencia* de los campos, no *función* de los endpoints. Un agente que lea el metadata de discovery puede ver, en el mismo documento, que el endpoint de autorización es una ruta reservada que todavía no acepta flujos OAuth reales. El campo `_comment` lo hace explícito:

```json
{
  "_comment": "xergioalex.com has no protected APIs today. This document is published for agent-readiness compliance per RFC 8414 / OIDC Discovery 1.0. Endpoints are reserved paths; they do not currently accept real OAuth flows.",
  "issuer": "https://xergioalex.com",
  "authorization_endpoint": "https://xergioalex.com/oauth/authorize",
  "token_endpoint": "https://xergioalex.com/oauth/token",
  "jwks_uri": "https://xergioalex.com/.well-known/jwks.json",
  "grant_types_supported": ["authorization_code"],
  "response_types_supported": ["code"]
}
```

Para `/.well-known/oauth-protected-resource`, la forma honesta es una autoreferencia: *este recurso está gobernado por el propio servidor de autorización (stub) del sitio*. Es una tautología, pero una verdadera, y RFC 9728 lo permite.

```json
{
  "_comment": "xergioalex.com is a static content site with no OAuth-protected resources today...",
  "resource": "https://xergioalex.com",
  "authorization_servers": ["https://xergioalex.com"],
  "scopes_supported": [],
  "bearer_methods_supported": []
}
```

Si el sitio alguna vez tiene una API protegida real, estos documentos se vuelven el punto de partida, no un reemplazo.

## El API catalog

RFC 9727 define el API catalog como un documento `application/linkset+json` en `/.well-known/api-catalog`. Cada entrada del linkset se ancla en una URL base y tiene `links[]` con dos valores de rel requeridos: `service-desc` (URL del spec OpenAPI) y `service-doc` (URL de docs humanas).

El sitio no tenía spec OpenAPI para sus endpoints JSON. Así que escribí uno mínimo en `public/openapi.json` — un documento 3.1 cubriendo `/api/posts.json`, `/api/posts-en.json`, `/api/posts-es.json`, `/api/series/{lang}` y `/api/timeline/{lang}`. Más corto que un proyecto de fin de semana; los schemas se apoyan en `type: array` y `type: object` sin modelar exhaustivamente cada campo.

El catálogo mismo apunta `service-desc` al archivo OpenAPI y `service-doc` a `llms.txt` (que el sitio ya servía desde el trabajo anterior sobre Markdown for Agents):

```json
{
  "linkset": [
    {
      "anchor": "https://xergioalex.com/api/",
      "links": [
        { "rel": "service-desc", "href": "https://xergioalex.com/openapi.json", "type": "application/json" },
        { "rel": "service-doc", "href": "https://xergioalex.com/llms.txt", "type": "text/plain" }
      ]
    }
  ]
}
```

El archivo vive en `public/.well-known/api-catalog` — sin extensión, a propósito. El Content-Type es `application/linkset+json`, no el default `application/json`, así que agregué un override en `public/_headers`:

```text
/.well-known/api-catalog
  Content-Type: application/linkset+json
  Cache-Control: public, max-age=300, must-revalidate
```

Este detalle me costó un rebuild. La primera versión se sirvió como `application/json` y me sorprendió que el tablero la rechazara. El SKILL.md literalmente dice "NO `application/json`" en la sección de trampas. Lo pasé por alto. Lee tus specs.

## La MCP server card y el índice de skills

MCP Server Card (SEP-1649) vive en `/.well-known/mcp/server-card.json`. La estructura son cuatro campos — `serverInfo.name`, `serverInfo.version`, `transport.endpoint` y `capabilities[]`. El array `capabilities` toma valores string de `tools`, `resources`, `prompts` — más simple que el objeto anidado que había borrado inicialmente:

```json
{
  "serverInfo": {
    "name": "xergioalex-site",
    "version": "1.0.0",
    "description": "..."
  },
  "transport": {
    "endpoint": "https://xergioalex.com/"
  },
  "capabilities": ["tools", "resources"]
}
```

El índice de Agent Skills Discovery es más interesante. El RFC v0.2.0 de Cloudflare define `/.well-known/agent-skills/index.json` como un documento con un identificador `$schema` (opaco — no tiene que resolverse) y un array `skills[]` donde cada entrada nombra una skill, enlaza a un archivo SKILL.md e incluye un digest SHA-256 de los bytes servidos del archivo.

Podría haber escrito mis propios archivos SKILL.md. Por ahora, estoy en modo pointer — mi índice referencia los ocho archivos SKILL.md canónicos de Cloudflare en `isitagentready.com/.well-known/agent-skills/<skill>/SKILL.md`, con el digest calculado desde los bytes servidos por Cloudflare al momento de generación. Un script pequeño en `scripts/generate-agent-skills-index.mjs` trae cada URL, la hashea y escribe el índice:

```js
async function sha256OfUrl(url) {
  const res = await fetch(url);
  const bytes = new Uint8Array(await res.arrayBuffer());
  const hex = createHash('sha256').update(bytes).digest('hex');
  return `sha256:${hex}`;
}
```

La primera corrida del script produjo el digest equivocado. Había hasheado la representación local en memoria de la respuesta — pero el spec quiere los bytes *tal como se sirven*. Arreglarlo fue un cambio de una línea (usar `arrayBuffer()`, no un round-trip a través de un string con `text()`). El índice ahora tiene ocho entradas, cada una con un digest real que coincide con lo que sirve el CDN de Cloudflare.

El generador corre en `prebuild`, que conecté reemplazando un script placeholder en `package.json`:

```json
"prebuild": "npm run generate:agent-skills-index",
"generate:agent-skills-index": "node scripts/generate-agent-skills-index.mjs",
```

Cada build futuro regenera el índice desde bytes en vivo de Cloudflare. Si Cloudflare versiona un SKILL.md, el digest se actualiza en mi próximo deploy.

## WebMCP

La última revisión es la única que no es un archivo estático. WebMCP es una API de navegador — `navigator.modelContext.registerTool()` — y el tablero carga la página en un navegador sin cabeza para ver si tu sitio registra alguna herramienta al cargar.

El spec (y el SKILL.md) es específico sobre la forma: cada herramienta necesita `name`, `description`, un `inputSchema` que sea JSON Schema válido y un callback `execute`. Pasas una señal de `AbortController` para que el registro pueda revocarse cuando el componente se desmonta. No pasar la señal es una de las trampas listadas.

Escribí un componente Svelte 5 en `src/components/agent/WebMCPBridge.svelte`. No renderiza nada visible. Se monta en `client:idle` — hidratación de baja prioridad — detecta la API por feature, luego registra tres herramientas de solo lectura que mapean a endpoints que el sitio ya expone:

```svelte
<script lang="ts">
  import { onDestroy, onMount } from 'svelte';
  const { lang = 'en' }: { lang?: 'en' | 'es' } = $props();
  let controller: AbortController | null = null;

  onMount(() => {
    const mc = (navigator as any).modelContext;
    if (!mc?.registerTool) return;
    controller = new AbortController();
    const signal = controller.signal;

    mc.registerTool({
      name: 'search_blog',
      description: 'Search xergioalex.com blog posts by keyword.',
      inputSchema: {
        type: 'object',
        properties: { q: { type: 'string' }, lang: { type: 'string', enum: ['en', 'es'] } },
        required: ['q'],
      },
      execute: async ({ q, lang: l }) => {
        const endpoint = (l ?? lang) === 'es' ? '/api/posts-es.json' : '/api/posts-en.json';
        const posts = await (await fetch(endpoint)).json();
        const needle = q.toLowerCase();
        return posts.filter(p => p.title?.toLowerCase().includes(needle) || p.description?.toLowerCase().includes(needle)).slice(0, 20);
      },
    }, { signal });

    // ... list_series y open_post registrados igual
  });

  onDestroy(() => { controller?.abort(); controller = null; });
</script>
```

Tres herramientas, todas de solo lectura: `search_blog`, `list_series`, `open_post`. Sin escrituras, sin acciones destructivas, nada cross-origin. Los nombres de las herramientas coinciden conceptualmente con la lista `capabilities` de la MCP server card — aunque la server card solo anuncia las *categorías* (`["tools", "resources"]`), no las definiciones por herramienta.

Conectarlo al sitio es un import y un elemento, ambos en `MainLayout.astro`:

```astro
---
import WebMCPBridge from '@/components/agent/WebMCPBridge.svelte';
---

<!-- layout existente -->
<Footer lang={lang} />
<WebMCPBridge client:idle lang={lang as 'en' | 'es'} />
```

Cada página que use `MainLayout` — que es efectivamente todo el sitio público — ahora envía este componente. Para un agente cargando la homepage, tres herramientas están presentes en milisegundos de estar idle.

## Lo que el agente ve ahora

Apila los ocho artefactos. Cualquier agente de IA que le pegue a `xergioalex.com` ve, en orden:

1. Una respuesta HTML con un header `Link: </.well-known/api-catalog>; rel="api-catalog"`.
2. Un `robots.txt` declarando `Content-Signal: ai-train=no, search=yes, ai-input=yes`.
3. Un API catalog que apunta a un spec OpenAPI real y docs humanas.
4. Un documento de metadata de servidor de autorización OAuth — honesto, anotado, estructuralmente compatible.
5. Un documento de recurso protegido OAuth — autoreferencia honesta.
6. Una MCP server card declarando que el sitio soporta `tools` y `resources`.
7. Un índice de agent-skills con ocho entradas y digests SHA-256.
8. Una página de navegador que llama `navigator.modelContext.registerTool()` tres veces en idle.

Eso es una superficie de protocolo. Nada de esto era posible hace unas semanas — ninguno de los drafts anteriores existía como spec enviado todavía. Todo esto se ensambló justo encima de una preocupación sobre la que vengo escribiendo hace meses: que la medición del AEO iba años atrás de la optimización. Ya no.

## El 100

Envié las ocho brechas. El directorio `.well-known/` pasó de vacío a seis documentos, `robots.txt` ganó una directiva, `_headers` ganó tres reglas Link más overrides de Content-Type, y `MainLayout.astro` ganó un componente Svelte pequeño. El tablero, una vez que el sitio se redespliega, se mueve de 33 a 100.

Advertencias, dichas honestamente:

- `isitagentready.com` es el tablero de un vendor. Las definiciones pueden cambiar.
- Los endpoints OAuth son rutas reservadas; devuelven 404 si un agente realmente intenta autorizar. Eso es intencional según la política de honestidad, y está documentado en el metadata mismo.
- El spec de WebMCP sigue moviéndose. Si la superficie de la API cambia, el componente Svelte la sigue.

Lo que sobrevive a un cambio de tablero: los archivos `.well-known/`. Cada uno está atado a un estándar real de IETF o WHATWG. RFC 8288, RFC 9727, RFC 9728, RFC 8414 — esos no se van a ninguna parte. Incluso si la herramienta Agent Readiness desaparece mañana, las primitivas que envié siguen válidas.

## Lecciones

*Lección: la honestidad le gana al teatro.* Los documentos OAuth que apuntan a nada son mejores que los documentos OAuth que mienten. El campo `_comment` no es hacer trampa; es una forma compatible con el spec de decir "la forma está aquí; la función no". Los agentes pueden razonar sobre eso.

*Lección: el directorio `.well-known/` es solo JSON.* La mayor parte del trabajo es leer los ejemplos de SKILL.md y hacerlos coincidir tal cual. Rebuildéee dos veces — una por el Content-Type equivocado en el api-catalog, otra por la codificación SHA-256 equivocada en el índice de skills. Ambos fueron errores pequeños sobre los que el SKILL.md me había advertido en su lista de trampas. Lee tus specs, incluso cuando creas que ya sabes qué van a decir.

*Lección: alinea la server card y la superficie WebMCP.* La MCP server card anuncia `capabilities: ["tools", "resources"]`, y el componente WebMCP registra tres herramientas. Describen la misma superficie de agente en dos granularidades diferentes. Si alguna vez agregas una herramienta, agrégala en ambos lugares. Si alguna vez renombras una, renómbrala en ambos.

Si quieres el *por qué* detrás de cada primitiva — para qué sirve cada spec, cómo se ve un ejemplo mínimo válido, qué falla — ese tema da para una entrada aparte.
